### Q8：一人公司的法律实体选择——个体工商户、有限公司、一人有限责任公司各有什么陷阱？

**A：** 法律实体选择是 OPC 第一个重要决定，选错了后面要花大代价改。

**个体工商户（个体户）：**

*优点*：注册简单（几百块、几天搞定）；税务简单（综合税率低，很多地方有税收减免）；注销也快。
*陷阱*：无限责任——生意上的债务要用个人财产偿还。如果做 SaaS（可能有数据安全风险/客户索赔），风险敞口太大。AI 产品如果出了版权/数据问题，你个人连带。
*适合什么人*：纯自由职业（设计/翻译/咨询），没有产品责任风险。不适合做任何有品牌/产品/客户的业务。

**一人有限责任公司（法人独资有限公司）：**

*陷阱1*："一人有限"≠"有限责任"自动保护。公司法要求个人财产和公司财产严格分离。但很多 OPC 会把个人卡收公司款、公司支付个人消费→这就构成了"人格混同"，法院可以刺破公司面纱，让股东承担连带责任。
*陷阱2*：一人有限责任公司的年检/年报义务比普通有限公司更严格。且股东不能兼任监事（需要另外找一个人当监事）。
*陷阱3*：如果你未来要融资或接纳合伙人，一人有限责任公司的股权结构变更非常麻烦。不如一开始就注册成普通的多人有限公司（即使其他股东只占1%）。

**最推荐方案**：注册 **普通有限责任公司（多人）**，让你信得过的家人/朋友持有 1% 股份。这样：
- 享有有限责任保护
- 融资/合伙人进入更方便
- 年检要求正常
- 成本几乎一样（注册约￥500-1000）

**国际维度**：如果你的客户在海外：
- 注册美国公司（Delaware 或 Wyoming）在很多场景下更有优势
- 用 Stripe Atlas（$500 包括注册+银行开户+税务）
- 中国居民做海外业务要注意：外汇管制和税务申报（你仍可能被中国税务局视为税务居民）

### Q9：一人公司的税务和发票——在中国做 OPC 最容易被忽略的坑是什么？

**A：** 税务是 OPC 最容易被忽视但后果最严重的问题。以下四个坑最常见也最致命。

**坑1：个人微信/支付宝收款不报税**

很多 OPC 认为"客户转到我私人的微信/支付宝，没人查得到"。现实：
- 2023年起的"金税四期"已经打通了银行+税务+市场监管的数据
- 个人账户年流水超过一定金额（各地区不同，约 ￥50-100万）会自动触发银行反洗钱报告
- 大平台（微信/支付宝）也已接入税务数据报送
- 一旦被查到，补税+滞纳金+0.5-5倍罚款

**正确做法**：必须注册公司 → 开对公账户 → 开发票 → 走公账。

**坑2：开票类型选择错误**

- **增值税普通发票**：多数小微企业可以用，税率 1%（小规模纳税人）或 3%
- **增值税专用发票**：客户一般纳税人可以抵扣进项税。如果你的客户是企业（B2B），他们大概率需要专票。不给开专票=失去客户。
- 实操：大多数科技/软件 OPC 应该登记为小规模纳税人（年销售额≤500万），按1%交增值税，可自开或代开专票。

**坑3：跨境收入的税务处理**

卖给海外客户（美国/欧洲/SaaS订阅）：
- 你不需要开发票给海外客户（他们不需要中国的发票）
- 但你需要申报收入，在国内缴税
- 很多 OPC 自以为"美元收到境外银行或PayPal，中国税务局不知道"→ 这只是时间问题。PayPal 现在也会同步数据给中国税务机关
- 更安全的做法：正式办理税务登记，如实申报。实际税率（小微企业优惠后）可能比你想的低。

**坑4：缺乏记账和规划**

- 很多 OPC 一年到头只管开发，年底才想起记账
- 建议：每月花2小时做账，用每记/简道云/灵办等工具
- 把交 QQ（契税）、增值、所得税的时间点加入日历提醒
- 找一家靠谱的代账公司（每月 ￥200-500），即使你觉得"用不上"

### Q10：一个人写代码最容易犯的架构错误是什么？

**A：** 这是我从大量一人项目的失败中学到的教训。OPC 的代码风格和团队完全不同——最大的敌人不是代码质量问题，而是**"过度设计"**和**"完全不设计"**之间的摇摆。

**错误1：一上来就用微服务 / 复杂的架构**

OPC 的通病——项目刚开就拆分前端、后端、AI、数据库、队列、缓存、消息系统。结果：
- 开发速度极慢，3个月还没见 MVP
- 部署/运维复杂度超出承受能力
- 遇到 bug 要排查多个服务
- 最后一个人维护不了，重构又没钱

**正确做法**：**单体优先（Monolith First）**。一个 Python/FastAPI 或 Node/Express 项目跑所有功能。等真的有流量压力再拆。99% 的 OPC 项目永远不会到需要拆的规模。

**错误2：完全不写文档/注释**

一个人的代码只有自己在写的时候理解。3个月后回来看，连自己都看不懂了。典型的"写出只有神能维护的代码"。

**正确做法**：每次 commit 写清晰的信息（用 AI 帮忙写也没问题），API 用 OpenAPI 自动生成文档，数据库字段加注释。AI（Claude/Cursor）现在可以帮你做"反注释"——你写代码它自动加注释。

**错误3：不写测试**

"反正就我一个人，测什么测"——这是 OPC 最大的认知盲区。没有团队帮你发现 bug，你的每一次 bug 都直接面向客户。对 SaaS 产品来说，一个严重的 bug 可能让你失去一批用户。

**正确做法**：不要求100%覆盖率，但业务核心逻辑必须有测试。可以只用 AI 生成测试（写代码时让 Cursor 自动生成测试用例），不花额外时间。至少做到：支付流程、用户认证、数据导出这三个核心路径有测试覆盖。

**错误4：没有备份/迁移策略**

用 SQLite？开心。然后笔记本坏了。数据全丢。

**正确做法**：Day 1 就选好数据库并配置自动备份。对于 OPC 项目，推荐 Supabase（免费额度足够）或 PlanetScale（Serverless MySQL）。两者都自带备份和自动迁移工具。至少每天自动化备份到云端。

**错误5：技术栈过于冷门或太"潮"**

Bun、Deno、HTMX、Rust for web、Svelte 5——听起来很酷，但问题在于：
- 中文社区资源少，踩坑没人帮
- AI 代码助手对这些小众框架的训练数据不足，生成质量差
- 一年后你想加功能，冷门框架可能已经被弃坑

**正确做法**：选择 **"无聊但成熟"** 的技术栈。Next.js + PostgreSQL + Vercel 或者 FastAPI + SQLAlchemy + DockerCompose。无聊 = 稳定 = 社区大 = AI 训练数据多 = 运维省心。

### Q11：OPC 在客户合同中最容易被坑的地方是什么？

**A：** 很多一人公司创始人对合同的态度是"先做起来再说，合同后面补"——这是最常见的陷阱。

**陷阱1：没有书面合同**

口头协议，微信确认，然后：
- 客户拖款3个月
- 客户说你"承诺了额外功能"，不承认"额外收费"
- 客户拿着你的代码说"这是我的"
- 你没有任何证据证明原始约定

**解决方案**：即使是最简单的合作，也要有一个标准合同模板。Annie D. 是一个著名的 OPC SaaS 创始人，她的建议是：**"和客户签合同不是不信任，是尊重彼此的时间。"** 用 Notion 或 Google Docs 做一个模板，每次只需改日期+价格+交付物。

**陷阱2：知识产权归属不清**

最常见："我帮客户开发了定制的 AI 工具，但合同里没写 IP 归属。"
- 客户认为"付了钱，代码就是我的"
- 你认为"只授予了使用权，通用模块版权还是我的"
- 如果走法律途径，中国法院倾向保护实际出资方的利益，除非合同明确约定

**解决方案**：合同明确写三部分：
1. 通用模块（Framework/组件库）→ 你保留版权
2. 定制化功能（具体业务逻辑）→ 客户所有
3. 过渡期使用许可（客户拿到源码后你的使用范围）

**陷阱3：付款周期太长**

很多客户喜欢"项目完成后30天付款"。但 OPC 的现金流脆弱。一个30万的单子做完后等30天收款，假设过程中你没有其他收入，第20天可能就撑不住了。

**解决方案**：分期付款。至少做到：
- 30% 启动
- 30% 中期验收
- 30% 交付
- 10% 验收后

或者更好的方式：转成 SaaS 订阅——把合同从"一次性开发"改为"每月服务费"。

**陷阱4：无限修改条款**

"客户说还没满意，继续改"——没有明确定义"满意的标准"，你可以被无限的"再改一下"拖死。

**解决方案**：合同中明确迭代范围——"包含 3 轮修改反馈。超出部分按 ¥XXX/小时收费。" 或者限定修改范围——"修改仅限于已约定的功能规格范围内"。

## 第2批总结

法律实体选普通有限公司（不要个体户，不要一人有限公司）；税务必须正规化，金税四期下逃无可逃；代码走"单体优先+重要逻辑有测试+成熟技术栈"的路线；合同必须书面、必须明确 IP 归属和付款节点。OPC 的致命伤不是做不出产品——是做出来后被人起诉/被税务局追缴/自己和客户打官司，耗光了所有成本和精力。
