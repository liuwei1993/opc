### Q1894：一人公司如何用AI做Red Team（对抗性战略推演），具体操作步骤是什么？

**A：** Red Team在军事和大公司中是"模拟对手攻击你"的团队，OPC一人没法养一个专职团队，但AI可以扮演这个角色。核心操作分4步：

**第1步：构造Red Team Prompt框架。** 不要让AI"给我的产品提意见"——这太泛。正确做法是给AI一个完整的竞争对手画像："你现在是一家拿到$5M种子轮的创业公司CEO，你的产品和我完全相同，你有10人工程团队、$200K/月营销预算、6个月runway。你的目标是12个月内抢走我80%的客户。请列出你的攻击计划。" 这个Prompt会让AI从竞品视角输出：价格战策略（你能承受的最低价是多少？）、功能抄袭清单（你最容易被复制的3个功能）、渠道截流（你的获客渠道中哪个最容易被竞价截断）。

**第2步：量化脆弱点。** 对AI输出的每个攻击方向做3维评估：(1) 发生概率（0-100%，竞品是否真有动机这么做）；(2) 影响程度（如果发生，你损失多少%收入）；(3) 防御成本（你需要投入多少时间/金钱来防御）。风险分 = 概率 × 影响 / 防御成本。优先处理风险分最高的3项。

**第3步：执行"最小防御"。** OPC的防御不是建城墙，是让城墙看起来很高。具体操作：(1) 功能护城河——把AI列出的"最容易复制功能"加一层独特数据（用户行为数据、行业知识库），使复制成本从2周变成2个月；(2) 合同护城河——年付合同+数据迁移成本条款，让客户切换的实际成本>感知成本3倍；(3) 关系护城河——每季度1次1v1视频通话（前20个客户），AI无法替代的人际信任。

**第4步：季度复盘。** 每季度用AI重跑一次Red Team，但更换竞对画像（上次模拟VC-backed startup，这次模拟大公司新功能线，再下次模拟同层OPC价格战）。案例：某项目管理SaaS创始人每季度做此练习，Q2发现"Notion可能加甘特图"，提前3个月强化时间追踪差异化，当Notion确实发布甘特图时他已迁移到"建筑项目专用"定位，零流失。

**避坑：** (1) 不要让AI同时扮演Red Team和Blue Team——它会自洽，给你虚假安全感。分开两次会话，中间隔1天；(2) Red Team输出至少30%应该是你不愿听到的——如果全是"你很强不用担心"，说明Prompt不够尖锐，加一句"不要客气，越残酷越好"；(3) 不要每月做——频率太高会导致防御焦虑，季度足够。

### Q1895：Pre-mortem（预验尸）在OPC战略决策中如何落地？和传统SWOT分析比有什么优势？

**A：** Pre-mortem是Gary Klein提出的决策技术：假设项目已经失败了，倒推它为什么失败。比SWOT强在两点：(1) 强制你面对失败而非罗列优势（SWOT的"Strengths"部分经常变成自我安慰）；(2) 时间倒推比正向预测准确率高30%（Klein的原始研究数据）。

**OPC落地SOP（90分钟版本）：**

**阶段1：设定失败场景（10分钟）。** 选一个你正在做或即将做的决策（如"推新产品线"、"从月付转年付"、"进入日本市场"）。用AI生成失败叙事："假设现在是2027年6月，你的[决策]彻底失败了——产品下线、客户全部流失、你亏了$X。请写一份详细的'事故报告'，解释从2026年6月到2027年6月之间发生了什么导致这个结果。要求：至少包含5个具体事件节点，每个事件有时间、触发条件、你的反应。"

**阶段2：提取失败模式（20分钟）。** AI会输出一个时间线，你从中提取"失败模式"——通常是：(1) 现金流断裂（新产品烧钱但旧产品客户流失，两头空）；(2) 精力分散（一人管两条产品线，哪个都没做好）；(3) 市场时机错误（进入太早教育成本高、太晚竞争激烈）；(4) 技术债爆发（赶工上线→Bug频出→口碑崩塌）；(5) 关键依赖风险（某平台API变更、某渠道算法调整）。

**阶段3：计算防御ROI（30分钟）。** 对每个失败模式问3个问题：
- "这个事件发生的前兆信号是什么？"（如：月流失率从3%升到5%是现金流断裂的前兆）
- "我需要在什么时间点介入才能避免？"（如：流失率连续2个月>4%就暂停新产品投入）
- "最小防御动作是什么？"（如：新产品预算锁定在旧产品利润的30%，超了就停）

**阶段4：写入决策文件（30分钟）。** 把以上分析写成一个1页文档：决策内容 + 3个最可能的失败路径 + 每个路径的触发条件+防御动作。存入你的"决策日志"。6个月后回看，验证预测准确率——这本身就是Pre-mortem能力的训练。

**与SWOT对比数据：** 研究显示Pre-mortem使"过度自信偏差"降低40%（Prospective Hindsight研究，2012）。对OPC来说最大价值是：SWOT分析经常让你觉得"我有技术优势所以没问题"，而Pre-mortem会逼你面对"技术优势在渠道被切断时毫无用处"。案例：某SaaS创始人在进入欧洲市场前做Pre-mortem，发现"GDPR罚款"这个他从未想过的失败路径，花了$2K做合规审计，避免了潜在的€20M罚款风险。

**避坑：** (1) 不要把Pre-mortem变成"找借口不行动"——它的目的是找到防御方案后更自信地行动，而非阻止行动。如果Pre-mortem让你100%放弃，说明要么你的决策确实有问题，要么你的Pre-mortem不够具体（太抽象的恐惧不产生行动）；(2) 每个决策只做一次Pre-mortem——反复做会变成焦虑循环；(3) 和你的"一人董事会"分享Pre-mortem结果（见上批讨论），获取外部视角。

### Q1896：Feature Sunset（功能下线）在OPC场景下如何执行？什么样的数据信号告诉你"这个功能该杀了"？

**A：** 大公司杀功能需要走3个月审批流程，OPC的优势是你一个人说了算——但这也意味着你没有"政治缓冲"，杀错了没人帮你背锅。以下是数据驱动的功能下线决策框架：

**数据信号层（必须满足至少2个才考虑下线）：**

| 信号 | 阈值 | 数据来源 |
|------|------|----------|
| 使用率 | <5% MAU在过去30天触碰过 | PostHog/Mixpanel事件追踪 |
| 留存贡献 | 使用该功能的用户留存率 ≤ 未使用的 | Cohort分析，按功能使用分组 |
| 维护成本 | 每月>2小时Bug修复 + 每次部署需要回归测试 | 你自己的时间日志 |
| 支持负担 | 占总Support Ticket>10% | Help Scout/Intercom标签统计 |
| 新用户困惑 | 新用户首次Session在该功能停留>60秒但不操作 | Hotjar/FullStory回放 |

**关键洞察：使用率<5%不等于该杀。** 如果那5%是你最高价值客户（LTV前20%），这个功能是"VIP专属"，保留它但降低可见性。正确做法是"功能分层"：对普通用户隐藏，对高级用户保留。

**OPC执行SOP（30天流程）：**

**Day 1-7：数据确认。** 从PostHog拉取该功能的30天事件数据，交叉验证：(1) 绝对使用量（UV/PV）；(2) 使用者画像（付费用户vs免费用户，高LTVvs低LTV）；(3) 功能依赖关系（删了它会不会影响其他功能的数据流）。如果使用者全部是低LTV免费用户→果断杀；如果是高LTV付费用户→进入沟通阶段。

**Day 8-14：影响评估+通知。** 给使用该功能的用户发一封邮件（不需要公开公告，OPC规模不需要）：(1) 我们注意到你使用了[X功能]；(2) 我们计划在[日期]下线它，因为[原因]；(3) 替代方案是[Y]；(4) 如果你强烈需要保留，请回复这封邮件。回复率>30%=重新评估；<10%=继续执行。

**Day 15-21：渐进下线。** 不是一次性删除。第1步：从新用户注册流程中移除（不影响现有用户）；第2步：对现有用户加一个"此功能即将下线"的Banner + 替代方案引导；第3步：7天后完全关闭入口。

**Day 22-30：代码清理。** 从代码库中移除相关代码（不是注释掉，是删除）。更新文档。更新测试用例。记录在"功能墓地"文档中（为什么杀、什么时间、影响多少用户、有无回滚请求）——这是你未来的决策参考。

**避坑案例：** 某CRM产品杀了"邮件模板"功能（使用率3%），结果3个高LTV客户的整个工作流围绕它构建，集体流失。修复成本：重新上线+道歉+3个月免费。教训：使用率永远要和LTV交叉分析。另一个案例：某项目工具杀了"甘特图"（使用率8%），但发现它是最多被截图分享的功能——它是"展示给客户看的"功能，使用者少但影响力大。正确做法是保留但简化。

### Q1897：OPC如何低成本实现SBOM（软件物料清单）自动化？开源合规的"够用"标准是什么？

**A：** SBOM（Software Bill of Materials）在2024年后从"大企业才做的事"变成了"所有人应该做的事"——因为Log4j、xz-utils后门事件证明，你用的每个开源库都可能是定时炸弹。OPC不需要企业级方案，但需要一套$0成本+每月30分钟维护的最低可行方案。

**最小SBOM自动化栈（$0/月）：**

**第1层：依赖清单生成（自动）。** 
- Node.js项目：`npm ls --all --json > sbom.json` 或 用 `cyclonedx-npm`（OWASP官方工具，生成SPDX/CycloneDX标准格式）
- Python项目：`pip-licenses --format=json > sbom.json` 或 `safety check`（检查已知漏洞）
- Go项目：`govulncheck ./...`（官方漏洞扫描）
- 全语言通用：`syft`（Anchore开源工具）扫描Docker镜像生成完整SBOM

**第2层：许可证合规检查（CI/CD集成）。**
```yaml
# GitHub Actions示例
- name: License Check
  run: |
    npx license-checker --onlyAllow "MIT;ISC;BSD;Apache-2.0;BSD-2-Clause;BSD-3-Clause"
    if [ $? -ne 0 ]; then echo "❌ Forbidden license found" && exit 1; fi
```
这个在每次PR时运行，阻止任何非白名单License的依赖进入代码库。

**第3层：漏洞持续监控（被动）。**
- GitHub Dependabot（免费，自动PR修复）——配置`updates`频率为weekly
- Snyk Open Source（免费tier：200次测试/月）——比Dependabot更准，false positive率低40%
- Socket.dev（免费）——不仅检查漏洞，还检查"typosquatting"（依赖名拼写欺骗）和"protestware"（作者故意破坏的包）

**第4层：月度SBOM审计（30分钟/月）。** 每月1号花30分钟做：(1) 对比本月SBOM和上月SBOM的diff（新增/移除/升级了哪些依赖）；(2) 检查新增依赖的许可证是否在白名单内；(3) 检查是否有依赖的原作者已停止维护（>6个月无commit+无release）；(4) 对于核心依赖（你的产品直接依赖的，非传递依赖），检查是否有"License变更"新闻（如Elastic、HashiCorp案例）。

**OPC"够用"标准（80/20法则）：**

| 必须做 | 可以不做 |
|--------|----------|
| 白名单License（MIT/Apache/BSD/ISC） | 完整SPDX文档（除非客户要求） |
| CI/CD自动拦截非白名单依赖 | 实时SCA扫描（$500+/月方案） |
| Dependabot/Snyk漏洞告警 | 法律团队审查每个依赖（$300/小时） |
| 月度diff审计 | 每次部署都重新生成SBOM |
| 核心依赖替代方案预研 | 所有传递依赖都验证 |

**关键决策：传递依赖怎么处理？** 你的项目直接用了50个包，但传递依赖可能有500个。策略：(1) 只审计直接依赖的许可证（传递依赖许可证冲突风险极低且法律上你无直接责任）；(2) 漏洞扫描覆盖全部（包括传递依赖），因为漏洞不分层级；(3) 如果某个传递依赖是非白名单License，考虑：换掉直接依赖 > 联系上游改License > 接受风险（记录在决策文档中）。

**真实案例：** 2023年某SaaS OPC用了`event-stream`包（npm周下载量200万），该包被注入恶意代码窃取比特币钱包。如果该创始人有SBOM月度审计，会在diff中看到event-stream的version突然跳了一个major版本且新增了一个可疑依赖`flatmap-stream`。30秒搜索就能发现这是供应链攻击。成本：$0。避免的损失：客户数据泄露+声誉崩塌。

### Q1898：一人公司的BCP（业务连续性计划）测试频率应该是多少？如何在不中断业务的情况下执行？

**A：** 大公司的BCP测试是每季度全链路演练+每年大规模DR（灾难恢复），OPC做不到也不需要。但"从不测试"和"每季度全面测试"之间有大量中间地带。以下是按业务阶段递增的测试频率框架：

**阶段1：$0-5K MRR（生存期）—— 每半年1次Paper测试（2小时）**

Paper测试不需要任何技术操作：你坐下来，用笔和纸（或Notion）模拟"如果X发生了，我怎么做"。
- 场景A：你的主服务器被AWS/Terminated（账号误封）——你有备份吗？恢复需要多久？
- 场景B：你的域名被锁定（WHOIS信息过期）——你有Registrar的2FA吗？恢复联系方式更新了吗？
- 场景C：你生病了2周不能工作——产品还能自动运行吗？客户支持谁处理？
- 场景D：Stripe/PayPal冻结你的账号——你有备用收款渠道吗？

每个场景写下"我现在能做什么"和"我需要提前准备什么"。如果>3个场景你的答案是"不知道/没准备"，优先解决最可能发生的2个。

**阶段2：$5K-20K MRR（稳定期）—— 每季度1次技术测试（半天）**

技术测试是真正动手操作：
- **备份恢复测试**：从备份恢复到临时环境，验证数据完整性。关键指标：RPO（Recovery Point Objective）——你能接受丢失多少数据？如果备份是每天凌晨3点，下午2点崩溃意味着丢失11小时数据。对SaaS来说通常可接受，对支付系统不可接受。
- **DNS故障转移测试**：如果你的主域名DNS挂了，你有多快能切到备用？Cloudflare免费plan提供30秒TTL+Anycast，基本解决。测试方法：临时把TTL改到60秒，切到备用NS，验证服务可用，切回来。
- **第三方服务降级测试**：如果你的AI API提供商（OpenAI/Anthropic）宕机4小时，你的产品会怎样？测试方法：在staging环境模拟API超时，看你的fallback逻辑是否正确触发。

**阶段3：$20K+ MRR（规模期）—— 每月1次自动化测试+每季度1次全面演练**

自动化测试（CI/CD集成，零人工干预）：
```yaml
# 每周自动执行
- name: BCP Smoke Test
  run: |
    # 1. 备份完整性校验（checksum比对）
    aws s3 cp s3://backups/latest/dump.sql /tmp/ && md5sum /tmp/dump.sql
    # 2. 恢复到临时DB并运行migration
    docker run -d --name bcp-test postgres:15 && psql -h localhost -U postgres < /tmp/dump.sql
    # 3. 运行核心API测试
    curl -f https://staging.example.com/health || exit 1
    # 4. 清理
    docker rm -f bcp-test
```

季度全面演练（Game Day，4-8小时）：
- 选一个"最坏场景"全链路执行：主DB宕机→切换到read replica→通知客户→修复→切回→复盘
- 记录实际RTO（Recovery Time Objective）vs 你承诺给客户的SLA
- 如果实际RTO > SLA，你需要：(1) 降低SLA承诺 或 (2) 投入更多自动化

**测试不影响业务的关键策略：**

1. **在staging环境做，不在production做。** 如果staging和production差异>20%，先缩小差距（IaC化），否则测试结果不可信。
2. **选低峰期。** SaaS产品通常UTC 02:00-06:00是低峰（美洲睡觉+欧洲刚起床+亚洲下午）。
3. **渐进式Chaos。** 不要第一天就Inject"整个Region挂了"。先从"单个Pod重启"开始，验证auto-healing是否正常，再升级到"整个AZ不可用"。
4. **通知客户。** 计划内的维护窗口提前24小时通知（Status Page + Email），比突然宕机后解释好100倍。

**一人BCP最小投入产出比：**
- 投入2小时Paper测试 → 发现5个单点故障 → 修复后避免$50K+潜在损失
- 投入半天技术测试 → 发现备份恢复失败（常见！30%的公司备份从未成功恢复过） → 避免"假安全感"
- 投入每月30分钟自动化 → 持续验证 → 避免"上次测试是6个月前，环境已经变了"

**避坑：** (1) 不要因为"我还小不需要BCP"跳过——$0 MRR时Paper测试只需2小时，成本极低但不做的话一旦出事就是game over；(2) 测试完必须写1页复盘（发现了什么/修复了什么/下次测什么），不写=白测；(3) 最大的BCP风险不是技术而是"Key Person"——如果你（唯一的创始人） incapacitated 2周，谁能登录你的账号处理紧急事务？解决方案：Dead Man's Switch（如Google的Inactive Account Manager）+ 信任的人持有加密的1Password Emergency Kit。
